# Standalone Intelligent Traffic & Travel Time Predictor 
**Location**: Porto, Portugal 
**Core Technology**: XGBoost, Geographic Vector Math (Haversine), Flask

---

## 1. System Overview
This project transitions from an abstract traffic density proxy to a **physical, mathematically grounded Intelligent Transportation System**. 
Instead of measuring arbitrary "congestion counts", the system maps the city into a strict **1-square-kilometer geographic grid** and predicts the exact moving speed of traffic dynamically. Using basic physics formulas alongside Machine Learning, the system natively calculates total travel-time durations without relying on external live-traffic APIs like Google Maps.

---

## 2. Data Engineering & Pipeline (`build_1km_speed_model.py`)

### A. Physical Grid Construction
Traditional arbitrary longitude/latitude bins were discarded. Instead, we used the spatial bounds of Porto and applied the **Haversine Formula** (which measures distance across the curvature of the Earth). 
The city's bounding box was physically sliced into a mathematically precise spatial grid where every single discrete cell represents an area of ~1km x ~1km.

### B. Speed Calculus
The raw dataset consists of 1.7 million taxi trajectories, where each ping is recorded exactly 15 seconds apart.
The extraction pipeline processed over 80 million data points:
- Measured the physical Haversine distance in meters between `Point A` and `Point B`.
- Computed the localized speed using the formula: `Speed (km/h) = Distance / (15 sec / 3600)`.
- Speeds > 120 km/h were discarded as GPS hardware noise.
- Every valid speed reading was mapped to its specific 1km geographic cell.

### C. Target Aggregation
The dataset was aggregated chronologically (`group_by(['hour', 'day', 'lon_bin', 'lat_bin'])`). The final output for the Machine Learning model is predicting the **Mean (Average) Traffic Speed** for any requested 1km block dynamically.

## 3. Machine Learning Architecture
**Algorithm**: `XGBRegressor` (GPU-accelerated via `tree_method='hist'`)
We selected XGBoost due to its historically proven performance on structured, tabular spatiotemporal matrices.

### Hyperparameter Tuning & Regularization
A high capacity was required to understand the deep spatiotemporal behavior mapping, but over-memorization (overfitting) was a significant risk. We mitigated this via explicit structural regularization:
* `max_depth = 8`: Keeps trees from tunneling too deeply into anomaly conditions.
* `min_child_weight = 20`: Explicitly prevents the neural trees from branching unless statistically backed by a minimum mass of 20 distinct traffic events.
* `subsample = 0.8`: Drops 20% of data randomly per tree to establish robust forest diversity.

### Evaluation Metrics
We evaluated the algorithm utilizing classical regression validation constraints against a held-out test set:
- **Mean Absolute Error (MAE)**: Provides the pure physical variance constraint. The model is proven accurately reliable within a variance of ~6 km/h.
- **Root Mean Squared Error (RMSE)**: Severely penalizes large algorithmic outliers ensuring safe prediction thresholds. 
- **R-Squared Accuracy ($R^2$)**: Demonstrated high validation scores generally between 0.80 and 0.85, tracking the practical mathematical ceiling for human-generated traffic flow.

## 4. Web Implementation & Standalone Routing (`app.py`)
The Flask backend serves as the bridge between theoretical data science and the user-facing web interface.

### Decoupling from Google Maps Live Traffic
Previously, the application utilized the Google Maps API to fetch the predicted "Live Traffic ETA". We successfully decoupled the AI brain to make it a standalone system:
1. **Geometric Pathing:** The Python script pings Google *solely* to obtain the static geometry of a route's Polyline and the physical absolute `Distance` (in km) connecting Source to Destination.
2. **Speed Execution:** Our cached `joblib` XGBoost model predicts the exact moving speed spanning every 1km vector on that route for a given 24-hour cycle. 
3. **Physics Extraction:** Following the core axiom `Time = Distance / Speed`, the Flask API structurally divides the route's km length by the ML Model's predicted speed.

### Real-World Benefit
The final rendered UI dynamically broadcasts the AI's internal Estimated Time of Arrival (ETA) based strictly on historical AI intelligence rather than standard public APIs, completely showcasing our end-to-end Machine Learning execution.
